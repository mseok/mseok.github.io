Git의 개념과 설정

버전관리와 공동작업을 위한 Git 개념과 설정

Dec 21, 2020

이 게시물에서는 git의 개념과 기본적인 설정에 대해 소개하며, pro git을 참고하였다.

Git을 사용하는 이유

코딩을 하거나, 디자인을 하는 사람들 혹은 글을 쓰는 사람들에게 가장 중요한 것 중 하나가 작업물들을 제때 백업시켜놓는 것이다. 하지만 일일이 작업을 할 때마다 작업한 날짜와 내용을 적어놓고 폴더를 복사하여 새로운 작업을 시작하는 일련의 과정들은 많은 자원들을 소모하며 귀찮다. 여러 사람들이 참여하는 공동 프로젝트를 관리하는 경우 데이터 백업의 문제가 더욱 복잡해진다. 이런 문제들을 한 번에 해결하기 위한 프로그램이 버전 관리 시스템(Version Control System, VCS) 이고, git은 가장 많이 쓰이는 버전 관리 시스템 중 하나이다.

Git vs GitHub

Git과 GitHub는 전혀 다른 개념이기 때문에 혼동하면 안된다. Git은 앞서 말한 것처럼, local 환경에서 작업하는 것들을 아카이브화 해놓은 것이라고 생각하면 된다. 이 때, 작업에 대한 거의 모든 것, 이를 테면 작업을 한 사람, 날짜과 내용 모든 종류의 파일들을 추적하여 그 흔적들을 저장해둔다. 반면, GitHub는 단순히 Git에서 저장한 것들을 클라우드 서비스와 같이 올려놓아 저장하거나 공유할 수 있는 공간이다. 앞으로 알아볼 것들은 모두 Git에 관한 것이다.

Git 기초

Git의 세가지 상태

Figure 1에서는 Git이 어떤 식으로 관리되는지를 보여준다. Git에는 크게 세 가지 상태가 있다.

  • Committed: 데이터가 “로컬” 데이터베이스에 저장된 상태
  • Modified: 수정한 파일을 “로컬” 데이터베이스에 아직 commit하지 않은 상태
  • Staged: 현재 수정한 파일을 곧 commit할 것이라고 표시한 상태

.git directory는 프로젝트의 모든 것들이 저장되는 곳을 말한다. 각 버전의 프로젝트들이 snapshot 형태로 암호화(SHA-1 hash) 되어 저장되어 있고, 프로젝트를 저장할 때 모든 파일들 각각을 암호화해서 저장한다. Working directory의 경우 .git directory에 있는 여러 가지 버전들 중 특정 버전을 보여주는 것 뿐이고, 우리는 이를 그 버전이 checkout된 상태라고 부른다. Staging area는 .git directory에 있다. 이는 단순한 파일이며 곧 commit할 파일들에 대한 정보를 저장한다. 이에 관한 자세한 정보들은 별도의 게시물에서 다루도록 하겠다. Figure 1에서 설명하는 git이 기본적으로 하는 일의 순서는 다음과 같다.

  1. Working directory에서 파일을 수정한다.
  2. Staging Area에 파일을 Stage 해서 commit할 snapshot을 만든다. 모든 파일을 추가할 수도 있고 선택하여 추가할 수도 있다.
  3. Staging Area에 있는 파일들을 commit해서 .git directory에 영구적인 snapshot을 저장한다.

Git configuration

Git 또한 다른 프로그램들을 사용하는 것과 마찬가지로 설정들이 있다. Git은 아래와 같이 세 가지 종류의 configuration 파일을 사용하며, 이 세가지 파일들은 각각의 명령어로 읽고 쓸 수 있다.

  • /etc/gitconfig<-git config --system
  • ~/.gitconfig (~/.config/git/config)<-git config --global
  • .git/config<-git config

이 세 가지 파일들은 역순으로 우선순위가 높으며, 따라서 local git repository에서의 설정은 .git/config 의 설정이 다른 파일들에서 설정된 모든 설정을 덮어씌우게 된다. 다음은 몇 가지 기본적인 configuration들이다.

  • 사용자 정보

    • git config user.name your_username
    • git config user.email your_email

    위의 두 가지 명령어로 local 환경에서 사용자의 이름과 이메일을 설정한다. 이렇게 설정한 이름과 이메일은 후에 pull, push를 할 때 user authentication 과정에서 쓰이게 된다.

    간혹, git configuration이 되지 않았을 경우에 git을 사용하다 에러가 발생하면서 이름과 이메을 설정하라는 문구가 뜨는 경우가 있다. 이 때 공용서버를 사용할 경우 --global 옵션을 지양해야 한다. --global 옵션은 두 번째 우선도를 가진 ~/.gitconfig 에 저장되며, 이게 저장되는 순간 local git repository의 .git/config 파일에 이름과 이메일이 없어도 --global 옵션을 주어 저장된 이름과 이메일로 push 혹은 pull이 되기 때문에 누가 어떤 일을 하여 업데이트 한 것인지 구분하기가 어렵게 된다.

  • CRLF

    Carriage return (CR)과 Line Feed (LF)는 흔히 발생하는 에러는 아니다. 이는 Windows 에서 작업하던 파일을 Linux 기반의 운영체제로 옮겨서 작업할 때 혹은 그 반대 상황에서 발생할 수 있는 에러이다. 이는 개행(new line)이라는 것을 Windows(CRLF), Linux(LF)가 다르게 처리해서 생기는 문제인데, 이에 관해서는 이 글을 참고하자. 그래서 CRLF가 충돌하는 문제가 발생하는 경우 commit할 때만 CRLF를 LF로 바꿔서 처리하게 하도록

    • git config core autocrlf input

    라는 명령어를 통해 해결할 수 있다.

  • Color

    Git을 사용할 때 로그 혹은 working directory에서 변경 사항을 보고자 할 때 변경 사항들이 터미널 상에서 흰색으로 나타나는 경우들이 있다. 가독성을 높이기 위해서

    • git config color.ui true

    라는 명령어로 color를 입히면 변경 사항들에서 어떤 것들이 추가되었고 사라졌는지 더 직관적으로 구분할 수 있게 된다.

이러한 설정들은 git config --list 라는 명령을 실행하여 볼 수 있다.
